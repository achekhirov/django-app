## Структура папок проекта Django: назначение


В вашем проекте присутствуют три ключевые папки: `main`, `blog` и `portfolio`. Ниже — краткое пояснение, зачем они нужны и как обычно используются в Django‑проектах.


### 1. `portfolio/`


**Это корневая папка проекта Django.**


Что содержит:
- `settings.py` — основные настройки проекта (секретный ключ, подключённые приложения, базы данных, статические файлы и т. п.);
- `urls.py` — главный файл маршрутизации (URL‑конфиг), который определяет, какие URL ведут куда;
- `asgi.py` и `wsgi.py` — точки входа для серверов:
  - `asgi.py` — для асинхронных серверов (например, Uvicorn);
  - `wsgi.py` — для синхронных серверов (например, Gunicorn, Apache/mod_wsgi).
- `__init__.py` — делает папку пакетом Python.


**Роль:**
- Центральная точка конфигурации всего сайта.
- Определяет глобальные настройки и маршрутизацию.


---

### 2. `main/`


**Это Django‑приложение, отвечающее за основную функциональность сайта.**


Что может содержать:
- `models.py` — модели БД (если есть данные, специфичные для «основной» части сайта);
- `views.py` — обработчики HTTP‑запросов (например, главная страница, контакты, о себе);
- `urls.py` — локальные URL‑маршруты для этого приложения;
- `templates/` — HTML‑шаблоны, используемые в этом приложении;
- `static/` — статические файлы (CSS, JS, изображения), относящиеся к этому приложению;
- `admin.py` — настройка панели администратора для моделей этого приложения;
- `apps.py` — конфигурация приложения (обычно минимальная);
- `migrations/` — история изменений схемы БД для моделей этого приложения.


**Роль:**
- Реализует ключевые страницы сайта (например, главную, раздел «Обо мне», контакты).
- Может содержать общую логику, используемую другими приложениями.


---

### 3. `blog/`


**Это отдельное Django‑приложение для блога.**


Что обычно содержит:
- `models.py` — модели для статей, категорий, комментариев и т. п.;
- `views.py` — представления для списка статей, детальной страницы, архива и т. д.;
- `urls.py` — маршруты блога (например, `/blog/`, `/blog/post/123/`);
- `templates/blog/` — шаблоны страниц блога;
- `admin.py` — регистрация моделей в админ‑панели;
- `apps.py` — конфигурация приложения;
- `migrations/` — миграции для схемы блога.


**Роль:**
- Изолирует функциональность блога от остального сайта.
- Позволяет легко подключать/отключать блог, обновлять его независимо.



---

### Почему так, а не «всё в одном»?


Django использует **архитектуру «приложений»**:
- Каждое приложение отвечает за **узкую область функционала** (блог, магазин, форум и т. п.).
- Приложения можно **переиспользовать** в других проектах.
- Код остаётся **модульным и понятным** — легче поддерживать и расширять.



Даже если у вас «один сайт», разделение на приложения помогает:
- Чётко разграничить зоны ответственности (блог vs. основная часть).
- Упростить тестирование и отладку.
- Легко отключать/подключать функции (например, временно убрать блог).



---


### Как это работает вместе?


1. **Django стартует** через `wsgi.py`/`asgi.py` → загружает настройки из `portfolio/settings.py`.
2. В `settings.py` перечислены все приложения (`INSTALLED_APPS`), включая `main` и `blog`.
3. В `portfolio/urls.py` маршруты «подключают» URL‑конфиги приложений:
   ```python
   path('', include('main.urls')),      # главная часть сайта
   path('blog/', include('blog.urls')),  # блог
   ```
4. Каждое приложение обрабатывает свои URL и логику независимо.

---

### Итог

- **`portfolio/`** — ядро проекта (настройки, глобальная маршрутизация).
- **`main/`** — основная функциональность сайта (главная, контакты и т. п.).
- **`blog/`** — изолированная функциональность блога.

Такой подход соответствует лучшим практикам Django и делает проект масштабируемым.
